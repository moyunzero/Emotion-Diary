# 迭代路线图 2026（竞品校准 · 工程优先）

> **状态**：已确认（brainstorming 2026-06）  
> **定位**：母规范 — 指导 `openspec/changes/003`–`009` 的排期、依赖与非目标；**不替代**各任务 `SPEC.md` 的验收契约。  
> **竞品依据**：2026 年情绪日记 / AI 日记 / 关系类应用趋势（Daylio、Stoic、Finch、Rosebud、Connected、BeSeen 等）与心晴MO 差异化定位的对照结论。

---

## 1. Understanding Summary

1. **要做什么**：在竞品分析基础上，将迭代重心调整为 **工程健康优先**，并同步落地 **留存（A1–A4）** 与 **数据信任（B1–B4）** 的全部用户可感知能力。
2. **为什么**：先稳住同步、音频、性能与测试信心，再靠提醒/回访与备份/回收站建立「敢记、敢留、敢换机」的信任，避免在债未清时铺新赛道。
3. **为谁**：现有心晴MO 用户（私人关系情绪日记）；维持单人私密、非医疗边界。
4. **工程 E 本周期范围**（原五项中 **可观测性暂不做**）：
   - 同步与多端（登录合并、软删与云一致、换机恢复）
   - 音频链路（录音、pending 上传、多入口竞态）
   - 性能与稳定性（大列表、Insights、导出）
   - 测试与 CI（同步/存储/音频相关单测与回归清单）
   - ~~可观测性（Sentry 等）~~ → **本周期不做**
5. **留存 A（全做）**：A1 本地提醒、A2 回访轻入口、A3 周回顾触达、A4 记录门槛优化。
6. **数据信任 B**：B1 回收站 UI、B3 同步/恢复文案分流、B4 永久删除 + 墓碑分流；**B2 本地快照已取消**。
7. **节奏**：无硬 deadline；全职投入；按 SSD `openspec/changes/` 逐个任务合入。
8. **明确不做**：对话式 AI 教练、情侣共写、健康数据关联、主题定制；**第三方崩溃监控本周期也不做**。

## 2. Assumptions

| 假设 | 说明 |
|------|------|
| 监控替代 | 无 Sentry 阶段靠手工回归 + 单测 + 本地 `logger`；线上问题复现成本较高，接受该 trade-off |
| AI 形态 | 保持现有批处理 AI；不新增聊天 Tab |
| 多语言 / 端侧 AI | 未列为非目标，但本周期 **不主动排期**（除非某 SSD 任务强依赖） |
| B2 本地快照 | **已取消**（产品决定；`002` 阶段二不再实施） |
| 工程与功能并行 | 全职下可波次并行，但 **每波须可独立验收、可回滚** |
| 性能基线 | Insights 首开 &lt; 2s（中端机、冷启动后首次进 Tab）为 **假设目标**，波 5 用实测写入 VERIFICATION |
| Web 通知 | A1 在 Web 可降级（无本地通知则仅 App 端完整能力） |

---

## 3. 战略一句话

**先以 SSD 任务稳住同步、音频、性能与测试信心（E，不含监控），同步交付数据信任链路（回收站 → 永久删除 → 操作分流），最后在稳定基座上落地留存触达（提醒、回访、周回顾、更短记录路径）；明确不做对话 AI、情侣共写、健康关联、主题皮肤、本地快照与第三方监控。**

## 4. 竞品校准与产品边界

### 4.1 心晴MO 差异化（维持，不转向竞品主战场）

| 维度 | 心晴MO | 主流竞品典型路径 |
|------|--------|------------------|
| 核心场景 | 人际/关系情绪私人记录 | 泛情绪（Daylio）或伴侣共写（Connected） |
| 表达隐喻 | 气象站、心灵花园、气话焚烧 | 像素年历、虚拟宠物、CBT 模板 |
| AI 形态 | 批处理洞察（预测、处方、回顾一句） | 对话式教练（Rosebud） |
| 数据哲学 | 离线优先 + 可选云同步 | 本地优先或强云 AI |
| 回顾闭环 | 周/月回顾图导出到相册 | 应用内图表或 PDF |

### 4.2 本周期非目标（Explicit Non-Goals）

- 对话式 AI 教练（独立聊天、长期记忆追问）
- 情侣/双人共写、冲突复盘工具
- 健康数据关联（睡眠、用药、生理周期等）
- 主题定制 / 深度个性化皮肤
- 第三方崩溃/错误监控（Sentry 等）

### 4.3 README「未来计划」的调整含义

| 原 README 项 | 本路线图 |
|--------------|----------|
| 更多数据分析维度 | 并入波 5 性能与 Insights；不单独铺大分析赛道 |
| 主题定制 | **不做** |
| 多语言 | 不主动排期 |
| 情绪提醒 | **做**（波 6 / 009，默认关） |
| 数据备份/恢复 | **做**（波 2–4 / 005、007，优先于主题与 i18n） |

## 5. 波次路线（推荐：双轨并行）

| 路线 | 思路 | 结论 |
|------|------|------|
| 严格瀑布 | E 全部 → B → A | 用户价值滞后，不推荐 |
| **双轨并行（采用）** | E 按风险切波；产品轨在数据层就绪处早交付薄片 | **采用** |
| 信任前置 | 先 B+A 再 E | 在脆弱 sync/音频上叠功能，返工风险高，不采用 |

### 波次地图

| 波次 | 主题 | 核心交付 | 对应 SSD |
|------|------|----------|----------|
| **波 0** | 回归地基 | 同步/软删/协调器单测；手工回归清单 | 003 |
| **波 1** | 同步可理解 | B3 入口与文案分流；同步状态可见；同步状态机小步重构 | 004 |
| **波 2** | 信任可见 | B1 回收站 UI | 005 |
| **波 3** | 音频可靠 | 退避重试、pending 失败态、协调器回归 | 006 |
| **波 4** | 永久删除 | B4；`002` 阶段二快照 **已取消** | 007 |
| **波 5** | 性能 | 大列表、Insights 懒加载、导出内存 | 008 |
| **波 6** | 留存触达 | A1–A4 | 009 |

**波 6 产品默认**：情绪提醒 **默认关**，首次开启须系统权限说明。

### 并行与脆弱区规则

- **`useAppStore` 同步路径**：同一时期 **最多一个** 活跃 SSD（004 → 007 顺序）。
- **`recordingCoordinator`**：仅 **006** 主改；其他任务禁止顺手修改。
- **006 ∥ 005** 允许；**007** 须在 004、005 合并后启动。
- 每个任务必备 `SPEC.md` + `VERIFICATION.md`；004、007 建议补 `PLAN.md`。
- **007** 须在 SPEC 中引用并 supersede `002` 阶段二段落，避免双文档漂移。

### README 更新时机

- **007、009** 合并时更新根 README「数据同步」「未来计划」。
- **003–006** 仅在必要时更新工程文档。

## 7. 验收标准与非功能约束

### 7.1 全局质量门禁（无第三方监控）

每个 SSD 任务合并前 **必须**：

| 门禁 | 要求 |
|------|------|
| CI | `yarn typecheck` → `yarn lint` → `yarn test` 全绿 |
| VERIFICATION | 记录实际命令；设备项写「已测 / 未测 + 原因」 |
| 脆弱区 | 触达 `useAppStore` 同步、`recordingCoordinator`、`audioSync` 时，须覆盖 §6 回归清单对应项 |
| 用户数据 | 无静默丢条、无未说明硬删 |
| 隐私 | 不上报日记正文/标签/音频；本周期不接 Sentry |

### 7.2 NFR

| 维度 | 目标 / 假设 |
|------|-------------|
| 规模 | 单用户 5k+ 条目、50+ pending 音频仍可用 |
| 性能 | 波 5 后主列表滚动无明显卡顿；Insights 首开目标见 §2 Assumptions |
| 可靠性 | 离线可记；同步失败可重试且可见；音频 pending 退避后仍失败须用户可见 |
| 隐私 | 云同步须 RLS + 可理解说明 |
| 多端 | iOS/Android 一致；Web 允许 A1 降级 |
| 可维护 | 一任务一主题 PR；禁止「顺手」改脆弱区外文件 |

### 7.3 各任务最低验收（摘要）

| 任务 | 必须验收 |
|------|----------|
| **003** | 合并同 id、软删过滤、coordinator `release` 单测；回归清单成文 |
| **004** | 三种数据操作可区分；同步失败非静默 |
| **005** | 软删 → 回收站 → 恢复 → 主列表；多设备场景写入 VERIFICATION |
| **006** | pending 退避可测；失败可重试；「编辑→关→记一笔录音」设备通过 |
| **007** | 永久删除与软删分流；回收站「永久删除」可演示 |
| **008** | 大列表 + Insights 基准；导出大月回顾不 OOM 或 documented 上限 |
| **009** | A1 默认关；A2–A4 各一条可演示路径 |

## 10. 实施 handoff

开始某一编号任务前：

1. 阅读本文件对应波次与依赖表。  
2. 自 `openspec/templates/` 创建 `changes/<编号>-<名称>/SPEC.md`。  
3. 对照 `engineering-quality.md` §2 脆弱区与 §2.3 录音约定。  
4. 实现后更新 `VERIFICATION.md`；行为变更同步领域规范与（若适用）根 README。

**建议下一任务**：真机回归清单 S1–S3、R1 收尾；或按发布节奏打 tag / PR 分批合入。
