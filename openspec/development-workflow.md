# 开发工作流文档

本文档定义了情绪日记应用的开发工作流程，基于 OpenSpec 规范驱动开发方法。

## 任务级规范（`openspec/changes/`）

与根目录各领域规范（`data-models.md`、`ui-components.md` 等）配合使用：

- **新功能 / 重要修复 / 跨模块重构**：在 `openspec/changes/<编号>-<简短名称>/` 下建立 `SPEC.md`；复杂任务再补 `PLAN.md`、`VERIFICATION.md`。模板见 `openspec/templates/`。
- **完整闭环、自审清单、与工程/领域文档分工**：见 **[openspec/README.md](./README.md)**；**「不要双写」**见 **`engineering-system.md` / `engineering-quality.md` 文首**。

工程约定与风险见 [`engineering-quality.md`](./engineering-quality.md)；栈、目录、外部集成见 [`engineering-system.md`](./engineering-system.md)。

## OpenSpec 工作流程

OpenSpec 采用规范驱动开发（Spec-driven Development，SDD）方法，通过在编写代码之前锁定意图，确保人类和 AI 在项目需求上达成一致。

### 工作流程概述

```
提案（Proposal） → 审查（Review） → 实施（Implementation） → 归档（Archive）
```

### 1. 提案（Proposal）

**目标**：创建变更提案，描述所需的规范更新。

**步骤**：
1. 确定变更需求（新功能、修改功能、修复问题）
2. 创建提案文档，描述：
   - 变更目标
   - 变更范围
   - 预期结果
   - 影响分析
3. 与团队成员和 AI 助手讨论提案
4. 根据反馈调整提案

**示例**：
- 添加新功能：在 `openspec/` 目录下创建提案文档
- 修改现有功能：更新相应的规范文档
- 修复问题：在规范文档中记录问题和解决方案

### 2. 审查（Review）

**目标**：与 AI 助手一起审查和完善提案，确保所有利益相关者达成一致。

**步骤**：
1. 审查提案的完整性和准确性
2. 检查提案是否符合项目规范
3. 评估变更的影响范围
4. 与 AI 助手讨论实现细节
5. 确认提案可以实施

**要点**：
- 确保提案与现有规范一致
- 检查是否需要更新多个规范文档
- 评估技术可行性和复杂度
- 确认变更不会破坏现有功能

### 3. 实施（Implementation）

**目标**：根据批准的规范实施任务。

**步骤**：
1. 根据规范文档实现代码
2. 遵循项目的代码规范和最佳实践
3. 编写必要的测试（如适用）
4. 进行代码审查
5. 确保代码符合规范要求

**要点**：
- 严格按照规范文档实现
- 如果发现规范有误，先更新规范再实现
- 保持代码与规范同步
- 编写清晰的代码注释

### 4. 归档（Archive）

**目标**：将完成的变更归档，并将批准的更新合并回源规范。

**步骤**：
1. 更新规范文档（如果需要）
2. 记录变更历史
3. 更新项目文档（如 README）
4. 标记提案为已完成
5. 归档变更记录

**要点**：
- 确保规范文档与实际代码一致
- 记录重要的变更和决策
- 更新相关的文档和注释
- 保持规范文档的准确性

## 新功能开发流程

### 1. 需求分析

**步骤**：
1. 明确功能需求
2. 分析用户场景
3. 确定功能范围
4. 评估技术可行性

**输出**：
- 需求文档
- 用户场景描述
- 功能范围定义

### 2. 规范设计

**步骤**：
1. 更新相关规范文档：
   - 数据模型规范（如果需要新的数据模型）
   - UI 组件规范（如果需要新的组件）
   - 服务层规范（如果需要新的服务）
   - 工具函数规范（如果需要新的工具函数）
2. 设计 API 接口（如果需要）
3. 设计数据流
4. 设计用户交互

**输出**：
- 更新的规范文档
- API 设计文档
- 数据流图
- 交互设计文档

### 3. 代码实现

**步骤**：
1. 创建功能分支
2. 实现数据模型（如果需要）
3. 实现服务层（如果需要）
4. 实现 UI 组件
5. 集成功能
6. 编写测试（如适用）

**输出**：
- 功能代码
- 测试代码（如适用）
- 代码注释

### 4. 测试和验证

**步骤**：
1. 单元测试（如适用）
2. 集成测试（如适用）
3. 功能测试
4. 用户体验测试
5. 性能测试（如适用）

**输出**：
- 测试报告
- 问题列表
- 修复计划

### 5. 代码审查

**步骤**：
1. 自我审查
2. 同行审查
3. AI 辅助审查
4. 修复审查问题
5. 确认代码符合规范

**输出**：
- 审查报告
- 修复后的代码

### 6. 合并和部署

**步骤**：
1. 合并功能分支
2. 更新文档
3. 更新规范文档（如果需要）
4. 部署到测试环境
5. 部署到生产环境

**输出**：
- 合并后的代码
- 更新的文档
- 部署记录

## 现有功能修改流程

### 1. 问题识别

**步骤**：
1. 识别需要修改的功能
2. 确定修改原因（bug、优化、需求变更）
3. 分析影响范围
4. 评估修改复杂度

**输出**：
- 问题描述
- 影响分析
- 复杂度评估

### 2. 规范更新

**步骤**：
1. 查看相关规范文档
2. 确定需要更新的规范
3. 更新规范文档
4. 确认规范变更

**输出**：
- 更新的规范文档
- 变更说明

### 3. 代码修改

**步骤**：
1. 创建修改分支
2. 根据更新的规范修改代码
3. 确保向后兼容（如果需要）
4. 更新相关代码
5. 编写测试（如适用）

**输出**：
- 修改后的代码
- 测试代码（如适用）

### 4. 测试和验证

**步骤**：
1. 回归测试
2. 功能测试
3. 兼容性测试（如果需要）
4. 性能测试（如果需要）

**输出**：
- 测试报告
- 验证结果

### 5. 代码审查

**步骤**：
1. 自我审查
2. 同行审查
3. 确认修改符合规范
4. 修复审查问题

**输出**：
- 审查报告
- 修复后的代码

### 6. 合并和部署

**步骤**：
1. 合并修改分支
2. 更新文档
3. 更新规范文档（如果需要）
4. 部署到测试环境
5. 部署到生产环境

**输出**：
- 合并后的代码
- 更新的文档
- 部署记录

## 代码审查检查清单

### 规范性检查

- [ ] 代码符合项目规范
- [ ] 代码符合 TypeScript 类型规范
- [ ] 代码符合 ESLint 规范
- [ ] 代码注释清晰完整
- [ ] 变量和函数命名规范

### 功能检查

- [ ] 功能实现完整
- [ ] 功能符合需求
- [ ] 功能符合规范文档
- [ ] 错误处理完善
- [ ] 边界情况处理完善

### 性能检查

- [ ] 代码性能合理
- [ ] 避免不必要的渲染
- [ ] 使用适当的优化策略
- [ ] 内存使用合理
- [ ] 网络请求优化（如适用）

### 安全性检查

- [ ] 输入验证完善
- [ ] 输出编码正确
- [ ] 敏感信息保护
- [ ] 权限控制正确
- [ ] 错误信息不泄露敏感信息

### 可维护性检查

- [ ] 代码结构清晰
- [ ] 代码可读性好
- [ ] 代码可扩展性好
- [ ] 代码可测试性好
- [ ] 代码与规范文档一致

### 兼容性检查

- [ ] 向后兼容（如需要）
- [ ] 跨平台兼容（iOS、Android、Web）
- [ ] 浏览器兼容（Web）
- [ ] 版本兼容

## 与 AI 工具协作

### 使用 Cursor 开发

1. **查看规范文档**：
   - 在开始编码前，先查看相关的规范文档
   - 确保理解需求和设计

2. **引用规范文档**：
   - 在提示中引用相关规范文档
   - 要求 AI 助手按照规范实现

3. **代码生成**：
   - 使用规范文档作为上下文
   - 要求 AI 助手生成符合规范的代码

4. **代码审查**：
   - 使用 AI 助手审查代码
   - 检查代码是否符合规范

5. **规范更新**：
   - 如果发现规范有误，先更新规范
   - 再要求 AI 助手根据新规范实现

### 使用规范文档的优势

- **准确性**：AI 助手可以更准确地理解需求
- **一致性**：确保代码与规范一致
- **效率**：减少反复沟通和修改
- **质量**：提高代码质量和可维护性

## 版本控制

### 分支策略

- **main/master**：主分支，用于生产环境
- **develop**：开发分支，用于集成开发
- **feature/**：功能分支，用于新功能开发
- **bugfix/**：修复分支，用于 bug 修复
- **hotfix/**：热修复分支，用于紧急修复

### 提交规范

- 提交信息清晰描述变更内容
- 关联相关的规范文档
- 引用相关的 Issue 或 PR
- 遵循项目的提交规范

### 合并规范

- 代码审查通过后合并
- 确保规范文档已更新
- 确保测试通过
- 确保文档已更新

## 持续改进

### 规范文档维护

- 定期审查规范文档
- 及时更新规范文档
- 保持规范文档与实际代码一致
- 收集反馈并改进规范

### 流程优化

- 定期回顾开发流程
- 识别流程中的问题
- 优化流程以提高效率
- 分享最佳实践

### 团队协作

- 定期团队会议
- 分享经验和知识
- 讨论规范和流程改进
- 确保团队成员理解规范
