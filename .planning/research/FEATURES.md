# Feature Landscape

**Domain:** v1.1 工程重构与代码治理（增量棕地治理）
**Researched:** 2026-03-21

## Table Stakes

这些能力属于“工程治理里程碑的基础盘”。缺失会直接导致重构风险高、迭代变慢、回归不可控。

| Feature | Why Expected | Complexity | Notes |
| --------- | -------------- | ------------ | ------- |
| 超大文件拆分（优先 `app/profile.tsx`、`store/useAppStore.ts`、500+ 行核心组件） | 现状已识别为维护痛点；不拆分会持续放大变更冲击面 | High | 以“行为不变”为硬约束，按 feature 边界拆，不按技术层硬切 |
| 目录边界治理（feature/shared 分层归位） | v1.1 明确目标是降低跨层耦合与重复实现 | High | 先定义边界规则，再迁移代码；避免一次性大迁移 |
| 重复逻辑收敛（响应式、时间区间、统计辅助等） | 重复实现会造成修一处坏多处，阻碍后续演进 | Medium | 统一为单一工具入口，并给替换清单 |
| 死代码与无效导出清理 | 降噪是重构前置条件，可减少误读与无效依赖 | Medium | 先做“可证实未引用”清理，再做结构调整 |
| 测试治理（删示例/重复，保关键路径并补重构保护） | 治理后需要安全网；仅“有测试”但不聚焦关键路径价值低 | Medium | 关键路径至少覆盖：记录/洞察/导出/AI 兜底/同步核心流程 |
| 结构与规范基线（`STRUCTURE.md`、`.editorconfig`、命名约定） | 没有统一约束，重构后会快速回退到混乱状态 | Low | 文档与规则要与目录结构同步更新 |

## Differentiators

这些能力不是“是否能做治理”的门槛，但会显著提升 v1.1 的长期收益和团队协作效率。

| Feature | Value Proposition | Complexity | Notes |
| --------- | ------------------- | ------------ | ------- |
| 治理任务清单化 + 风险分级（低风险清理 -> 结构重构 -> 测试补强） | 把抽象“重构”变成可排期、可验收的工程工作包 | Medium | 每项输出“改动范围、回滚点、验证点” |
| 变更影响面地图（模块依赖与调用链可视化） | 让新手也能判断“改这里会影响哪里”，减少盲改 | Medium | 先覆盖高频改动区（profile/store/export/insights） |
| 重构验收门禁（最小 CI 质量门槛：类型、关键测试、lint） | 将质量从“人肉把关”变为“流程把关” | Medium | 先轻量门禁，避免一次性上过重流程拖慢开发 |
| 渐进式 PR 模板（目的/风险/验证/回滚） | 降低沟通成本，确保每次重构可追踪、可复盘 | Low | 与现有 `.planning` 流程打通，形成固定节奏 |

## Anti-Features

本里程碑应明确不做的事项，防止范围蔓延导致治理失败。

| Anti-Feature | Why Avoid | What to Do Instead |
| -------------- | ----------- | ------------------- |
| 新增业务功能模块（新洞察页、大型 UI 改版、新增长流程） | 会稀释治理投入，且引入新变量，难以判断回归来源 | 仅做“行为不变”的工程治理；新功能移入后续产品里程碑 |
| 一次性全仓大爆炸重构 | 回归与回滚成本过高，验证链路断裂 | 采用分批次增量重构，每批都可独立验证与回退 |
| 为“架构美观”强行引入重框架/重依赖 | 与“维护成本可控”目标冲突，短期收益小于迁移成本 | 优先在 Expo SDK 54 / RN 0.81 现有栈内收敛 |
| 追求 100% 测试覆盖率作为里程碑目标 | 会把精力从高风险路径转移到低价值覆盖 | 以关键路径保护和回归风险降低为测试目标 |
| 重写 AI/导出/同步等已稳定链路 | 高风险低收益，容易破坏现有用户价值 | 只做边界清晰化与重复逻辑收敛，不重写业务语义 |

## Feature Dependencies

```text
结构与规范基线
  -> 死代码清理
  -> 重复逻辑收敛
  -> 目录边界治理
  -> 超大文件拆分
  -> 测试治理（补重构保护）
  -> 重构验收门禁（CI 最小质量门槛）

治理任务清单化 + 风险分级
  -> 渐进式 PR 模板
  -> 变更影响面地图
```

## MVP Recommendation

Prioritize:

1. 结构与规范基线 + 治理任务清单化（先把“怎么做”标准化）
2. 死代码清理 + 重复逻辑收敛（先降噪，再动结构）
3. 目录边界治理 + 超大文件拆分（按高收益模块分批推进）
4. 测试治理 + 最小 CI 门禁（为后续重构提供持续保护）

Defer: 全量依赖升级与重框架迁移：超出 v1.1“治理不改产品价值”的目标边界，建议在 v1.2 另立专项评估。
